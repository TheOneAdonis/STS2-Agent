# STS2 AI Agent Mod - 实施计划 v2

## 目标

构建一个可运行的 `Slay the Spire 2` 原生 Mod，通过本地 HTTP API 暴露游戏状态和动作能力，再由 Python MCP Server 封装为 MCP 工具，供 LLM 自主完成整局游戏。

本版计划的原则：

- 先做最小纵切，再逐步扩展能力。
- 先冻结协议，再写实现，避免 C# 与 Python 双端反复返工。
- 每个阶段必须有可演示结果，不接受“代码先堆着，后面再联调”。
- 所有新增能力都必须补齐日志、错误返回和验收方法。

---

## 当前确认的技术前提

| 项目 | 实际值 |
|------|-------|
| 游戏引擎 | Godot 4.5.1 |
| 运行时 | .NET 9.0 |
| 主程序集 | `sts2.dll` |
| Mod 形态 | `.dll + .pck` 放入 `mods/` |
| 计划通信方式 | Mod 内本地 HTTP API + Python FastMCP |
| 目标平台 | Windows 本机开发与验证 |

## 阶段详情

### Phase 0A: 环境与运行链路确认

**目标**

确认本机具备完整开发和验证条件，并跑通最小 Mod 安装链路。

**任务**

1. 安装并验证 `.NET 9 SDK`、`Godot 4.5.1 Mono`、`Python 3.11+`、`uv`、`ILSpy`。
2. 确认游戏目录、`mods/` 目录、日志目录、游戏启动方式。
3. 验证最小 Mod 的放置方式和加载表现。
4. 记录所有真实路径、版本和操作步骤。

**交付物**

- `docs/setup.md`
- 环境检查清单
- 最小 Mod 加载截图或日志片段

**验收标准**

- 能完成“构建 Mod -> 安装到 `mods/` -> 启动游戏 -> 看到 Mod 日志”。
- 新人可仅按 `docs/setup.md` 复现一遍。

---

### Phase 0B: 逆向侦察

**目标**

建立后续实现所需的真实类型表和入口表，避免靠猜。

**任务**

1. 反编译 `sts2.dll` 并输出到本地目录。
2. 定位以下对象的真实类型、命名空间、关键字段、关键方法：
   - 运行管理
   - 战斗管理
   - 玩家对象
   - 手牌/牌堆
   - 怪物 Intent
   - 地图节点
   - 奖励屏幕
   - 事件选项
   - 商店对象
   - 场景或 screen 枚举
3. 记录单例入口、缓存点和可能需要 Harmony patch 的方法。
4. 标注高风险成员：`private`、`internal`、动态生成或生命周期不稳定的对象。

**交付物**

- `docs/reverse-engineering.md`
- 核心类型索引表
- 风险成员清单

**验收标准**

- 后续实现涉及的核心对象都能给出准确类型名和访问路径。
- 不再依赖“可能是这个类”这种模糊判断。

---

### Phase 1A: 协议冻结

**目标**

在实现前冻结 Mod 与 MCP 间的数据契约。

**任务**

1. 定义以下端点的请求响应结构：
   - `GET /health`
   - `GET /state`
   - `GET /actions/available`
   - `POST /action`
2. 定义统一错误格式：
   - `code`
   - `message`
   - `details`
   - `retryable`
3. 定义最小状态模型：
   - `screen`
   - `run_id`
   - `state_version`
   - `available_actions`
4. 定义动作请求模型：
   - `action`
   - 动作参数
   - 幂等和非法参数行为
5. 约定动作返回结构：
   - 是否成功
   - 错误信息
   - 执行后是否已稳定
   - 可选最新状态快照

**交付物**

- `docs/api.md` 初版

**验收标准**

- Python 客户端可只依据 `api.md` 编写，不需要查看 C# 源码。
- 后续新增字段遵循向后兼容原则。

---

### Phase 1B: Mod 骨架与日志

**目标**

搭建可加载的 Mod 骨架，并提供稳定的基础 HTTP 服务。

**任务**

1. 创建 `STS2AIAgent.csproj`。
2. 实现 `ModEntry.cs` 入口。
3. 实现最小 `HttpServer`、路由分发和 JSON 工具。
4. 建立统一日志规范：
   - 固定前缀
   - 请求 ID
   - 错误堆栈
   - 超时日志
5. 提供 `GET /health`。

**交付物**

- `STS2AIAgent/` 初始项目骨架
- `/health` 路由
- 基础日志能力

**验收标准**

- 游戏启动后日志出现 `STS2 AI Agent ready`。
- `curl http://localhost:8080/health` 返回 `200` 和版本信息。

---

### Phase 1C: 最小纵切

**目标**

尽快打通“游戏 -> Mod -> HTTP -> MCP -> 客户端”的最小闭环。

**任务**

1. `GET /state` 先只返回极简字段：
   - `screen`
   - `run_id`
   - `in_combat`
   - `turn`（若可获取）
2. `POST /action` 先实现一个最小动作：
   - 优先 `end_turn`
   - 如果风险更低，可先做内部 `noop` 作为链路验证
3. 实现 Python `sts2_client.py`。
4. 实现 MCP 工具：
   - `get_game_state`
   - 一个最小动作工具

**交付物**

- 可工作的最小 `GET /state`
- 可工作的最小 `POST /action`
- `mcp_server/server.py` 最小版本

**验收标准**

- MCP 客户端可读取当前状态。
- MCP 客户端可成功发起一次最小动作调用并得到结果。

---

### Phase 2: 战斗状态提取

**目标**

稳定输出战斗中的核心信息，先覆盖高价值字段，再扩展细节。

**任务**

1. 实现 `player` 子结构：
   - `hp`
   - `max_hp`
   - `block`
   - `energy`
   - 关键 powers
2. 实现 `hand` 子结构：
   - `index`
   - `id`
   - `name`
   - `cost`
   - `is_playable`
3. 实现 `enemies` 子结构：
   - `index`
   - `id`
   - `hp`
   - `block`
   - `intent`
   - `intent_damage`
4. 实现牌堆计数：
   - 抽牌堆
   - 弃牌堆
   - 消耗堆
5. 将复杂字段设为第二优先级：
   - 精确伤害预览
   - 动态费用细节
   - 完整 power 元数据

**交付物**

- `State/CombatStateExtractor.cs`
- 战斗态 JSON 文档更新

**验收标准**

- 连续多次读取同一战斗状态，关键字段稳定。
- 关键字段与游戏 UI 对照无明显偏差。

---

### Phase 3: 战斗动作执行

**目标**

稳定执行战斗动作，并清晰返回成功、失败和等待状态。

**任务**

1. 优先实现动作顺序：
   - `end_turn`
   - `play_card` 无目标
   - `play_card` 有目标
   - `use_potion`
2. 加入动作前校验：
   - 当前 screen
   - 当前回合
   - 索引有效
   - 目标有效
   - 资源满足
3. 定义“状态稳定”判断规则，至少明确：
   - 队列是否清空
   - 动画是否结束
   - 是否重新允许玩家输入
4. 为所有动作补齐失败返回和超时处理。

**交付物**

- `Actions/CombatActions.cs`
- `Actions/ActionExecutor.cs`
- 状态稳定等待机制

**验收标准**

- 每个动作都有成功与失败两条可验证路径。
- 非法动作不会导致请求悬挂或游戏卡死。

**验收标准**

## 统一实现约束

### 1. 协议约束

- 任何新增 API 字段都必须同步更新 `docs/api.md`。
- MCP 工具名、参数名、返回结构必须和 HTTP 协议一一对应。
- 错误返回必须机器可读，不能只写日志不返回。

### 2. 日志与诊断

- 所有 HTTP 请求必须有请求 ID。
- 动作执行必须记录开始、结束、结果、耗时。
- 逆向访问失败时必须输出类型名、成员名和异常。

### 3. 状态稳定性

- 所有动作都必须定义完成条件。
- 无法确认稳定时，返回明确的 `pending` 或超时错误，而不是假装成功。

### 4. 版本兼容

- Mod 启动时记录检测到的游戏版本。
- 若版本不匹配已验证范围，记录警告并允许进入受限模式。
